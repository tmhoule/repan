Foundations

What is "AI" — really?

For our purposes, modern AI is a large language model (LLM) — a statistical model trained on enormous amounts of text and code that predicts the next token given some context.

Pattern, not magic

Compresses billions of examples into weights. Output is shaped by your prompt + retrieved context.

Code-fluent

Trained on huge corpora of source code → strong at refactors, scaffolds, tests, explanations.

Tool-using

Modern models call tools, read files, run shell commands — they act, not just chat.

"AI" in this talk = an LLM you talk to from inside your editor, backed by a server we host on the lab network.

How it fits together

Three components, one pipeline

→

2. Language Model

The LLM itself — Claude, GPT, Llama, etc. Receives prompts, returns tokens.

3. AI Processing Computer

Good to know

The context window

Every request has a size limit — measured in tokens (roughly ¾ of a word). Everything you send and everything the model writes back has to fit. The model has no memory — your tool re-sends the entire conversation on every turn, which is why it fills up.

It belongs to the model

Claude ~200k tokens. GPT-4o ~128k. Set by whichever model you pick — not by LiteLLM, Continue, or OpenCode.

LiteLLM just passes through

It forwards your request to the model. No trimming, no magic. Same limit as calling the model directly.

Each turn replays everything

System prompt + every prior message gets re-sent. In OpenCode that includes every file read and every command output — long sessions fill fast.

Practical take: bigger window ≠ better answer. One task = one session. Point the AI at specific files instead of "read the repo." If answers start drifting mid-session, start fresh.

Step 0 · Required for everything

Get your LiteLLM API key

1

Open https://litellm-dev.net.mit.edu/ui in your browser.

3

Go to Virtual Keys → + Create New Key. Give it a name (e.g. vscode-continue), pick the model(s) you need.

Step 2 · Configure

Point Continue at LiteLLM

In the Continue panel, click the gear / model selector → Local Config. This opens ~/.continue/config.yaml. Paste this template and replace the key.

name: Local Assistant
version: 1.0.0
schema: v1

models:
  - name: Claude Sonnet (LiteLLM)
    provider: openai
    model: claude-sonnet-4-6
    apiBase: https://litellm-dev.net.mit.edu/v1
    apiKey: sk-REPLACE_WITH_YOUR_LITELLM_KEY
    roles:
      - chat
      - edit
      - apply

  - name: GPT-4o (LiteLLM)
    provider: openai
    model: gpt-4o
    apiBase: https://litellm-dev.net.mit.edu/v1
    apiKey: sk-REPLACE_WITH_YOUR_LITELLM_KEY
    roles:
      - chat
      - edit

context:
  - provider: code
  - provider: docs
  - provider: diff
  - provider: terminal
  - provider: codebase
Save the file. Continue reloads automatically. Pick a model from the dropdown at the bottom of the chat panel and send a test message.

Continue · context providers

What does @ do?

Typing @something in Continue's chat tells it to fetch real context and paste it into your prompt before sending to the model. The model answers with your actual code in front of it — not a guess from training data.

@codebase

Semantic search over the whole repo. Pulls the most relevant snippets. Best for "where / how does…".

@file

Attach one specific file. Faster and cheaper than @codebase when you already know where to look.

@diff

Your current uncommitted git diff. Great for "review my changes" or "draft a commit message."

@terminal

Recent terminal output. Paste a stack trace into context without copy-paste gymnastics.

@docs

Indexed external doc sites you've added. Ask questions against framework docs, not stale memory.

@url

Fetch a web page on demand — RFCs, blog posts, internal wiki pages.

First use of @codebase on a new repo builds an embeddings index — takes a minute. After that it's instant and updates as files change.

Configure

Proxy & LiteLLM config

1 · Set the lab proxy (if your shell needs it)

# macOS / Linux — add to ~/.zshrc or ~/.bashrc
export HTTP_PROXY=http://net-proxy.net.mit.edu:8080
export HTTPS_PROXY=http://net-proxy.net.mit.edu:8080
export NO_PROXY=localhost,127.0.0.1,.net.mit.edu

# Windows PowerShell (current session)
$env:HTTP_PROXY  = "http://net-proxy.net.mit.edu:8080"
$env:HTTPS_PROXY = "http://net-proxy.net.mit.edu:8080"
$env:NO_PROXY    = "localhost,127.0.0.1,.net.mit.edu"

2 · Point OpenCode at LiteLLM

Edit ~/.config/opencode/opencode.json (create it if missing):

{
  "$schema": "https://opencode.ai/config.json",
  "provider": {
    "litellm": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "LiteLLM (LLAN)",
      "options": {
        "baseURL": "https://litellm-dev.net.mit.edu/v1",
        "apiKey": "sk-REPLACE_WITH_YOUR_LITELLM_KEY"
      },
      "models": {
        "claude-sonnet-4-6": { "name": "Claude Sonnet 4.6" },
        "gpt-4o":            { "name": "GPT-4o" }
      }
    }
  },
  "model": "litellm/claude-sonnet-4-6"
}

OpenCode in practice

Clone a repo, add a feature

OpenCode runs as an agent in your terminal — it can read files, run commands, edit code, and run your tests. A real workflow:

# 1 · Pull the code
git clone git@github.net.mit.edu:team/widget-service.git
cd widget-service
git checkout -b feat/csv-export

# 2 · Launch the agent in the repo
opencode

# 3 · Ask for the feature (in the OpenCode prompt)
> Read the codebase. Add a /widgets/export.csv endpoint
  that streams all widgets as CSV. Match the style of the
  existing /widgets/export.json route. Add a unit test and
  run the test suite.

It will

Map the repo, find the JSON route, draft the handler, wire the router, write a test, run pytest/go test/etc., and iterate on failures.

You will

Watch each tool call, approve edits, and review the final diff with git diff before committing.

Then

git commit, git push, open the PR. Optionally ask OpenCode to draft the PR description.

Don't know the git commands? Just ask. Launch opencode in an empty folder and say "Clone <repo URL> and check out a new branch called feat/csv-export." The agent will run the commands for you.

OpenCode in practice

Generate documentation

Documentation is the highest-leverage AI task: tedious for humans, mechanical for an LLM that can already read every file in your repo.

Useful prompts

  • "Generate a README.md with overview, setup, env vars, and a quickstart."
  • "Add docstrings to every public function in src/api/. Match the existing style."
  • "Write docs/architecture.md — modules, data flow, external deps. Include a Mermaid diagram."
  • "Produce an OpenAPI spec from the route handlers in routes/."
  • "Draft a runbook for the nightly ETL job — failure modes and recovery steps."

Why it works well

  • The agent reads the actual code, not your memory of it.
  • Output is consistent across files — same headings, same tone.
  • You go from blank page → editable draft in one turn.
Treat the output as a first draft. Verify claims about behavior — models will confidently describe code that doesn't exist.
